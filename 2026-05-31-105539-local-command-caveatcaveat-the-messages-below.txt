     Plan: Chinese Language Support for Journey Map

     Context

     The journey map app (journeymapcollect/Journey Map v2.html) is a
     single-file SPA used in a study with Chinese-speaking participants. All UI
     text is hardcoded in English. Participants need to fill in stage names,
     feelings, notes, and reflections — but the interface labels and
     instructions are all in English, creating friction. The fix is a
     lightweight in-page language toggle (EN ↔ 中文) that persists across
     sessions and flows through to the PDF via the existing Puppeteer save path.

     Approach

     Zero-dependency i18n using:
     1. A TRANSLATIONS flat-key dictionary (en/zh) embedded in the <script>
     block
     2. data-i18n / data-i18n-ph attributes on static HTML elements
     3. An applyLang() function that walks the DOM and updates text/placeholders
     4. state.lang to persist the choice in localStorage (auto-flows to PDF via
     existing Puppeteer path)
     5. A "中文 / EN" toggle button added to the .chrome toolbar

     Critical File

     - journeymapcollect/Journey Map v2.html — single file containing all HTML,
     CSS, and JS (~1974 lines)
     - server.js — no changes needed: Puppeteer already injects state into
     localStorage, so lang flows through automatically

     function t(key, vars) {
       const lang = (state && state.lang) || 'en';
       const dict = TRANSLATIONS[lang] || TRANSLATIONS['en'];
       let str = (key in dict) ? dict[key] : (TRANSLATIONS['en'][key] || key);
       if (vars) Object.keys(vars).forEach(k => { str = str.replace(new
     RegExp('\\{'+k+'\\}','g'), vars[k]); });

     3. Add toggleLang() + applyLang() functions

     function toggleLang() {

     function applyLang() {
       document.documentElement.lang = (state.lang === 'zh') ? 'zh-Hans' : 'en';
       document.title = t('page.title');
       document.querySelectorAll('[data-i18n]').forEach(el => el.textContent =
     t(el.dataset.i18n));
       document.querySelectorAll('[data-i18n-ph]').forEach(el => el.placeholder

       document.querySelectorAll('[data-i18n-aria]').forEach(el =>
     el.setAttribute('aria-label', t(el.dataset.i18nAria)));

> is ngrok tunnel better or cloud deployment better?                            

● For a research study, the choice comes down to one key question: do 
  participants fill the map during a supervised session (you're present), or 
  independently on their own time?

  - Supervised sessions → ngrok is better. Your laptop runs everything, data
  stays with you, Puppeteer works natively, and you only need the tunnel active
  during the session. The free tier's changing URL is fine — you just reshare it
  each session.
  - Unsupervised / participants go at their own pace → Cloud deployment is
  better. The server needs to be always-on, and you can't have it depend on your
  laptop.

  My recommendation: ngrok, specifically using localtunnel (no account, no
  install, completely free) or ngrok's free tier. The submissions folder stays
  on your machine where you want the data, Puppeteer needs zero changes, and
  you're up in under a minute.

  The only real downside of cloud deployment for this app is that the
  submissions/ directory (JSON + PDF) is on an ephemeral cloud disk — you'd need
  to add a download endpoint to retrieve your data, which is extra work.

     Plan: Public URL via localtunnel for Journey Map

     Context

     server.js currently binds only to localhost:3000, so participants on other
     computers cannot access the app. The researcher runs the server on their
     own laptop during study sessions; participants need a URL they can open
     from their own devices. The solution is localtunnel — a free, zero-signup
     npm package that creates a public HTTPS URL (e.g.
     https://journeymap.loca.lt) tunnelled to the local server. No cloud
     deployment needed; submissions still save to the local submissions/
     directory.

     Why localtunnel over ngrok:
     - No account or API key required
     - Install as a dev dependency (npm install --save-dev localtunnel) — no
     separate binary
     - Supports a fixed --subdomain flag so the URL is stable across sessions
     (e.g. https://journeymap.loca.lt) as long as the subdomain isn't taken

     Puppeteer note: PDF generation already uses http://localhost:${PORT}/
     internally (line 75 of server.js), which bypasses the tunnel entirely —
     this keeps working without any change.

     - journeymapcollect/package.json — add localtunnel as a dev dependency
